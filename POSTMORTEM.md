> **用途**: 记录每次发布过程中发现的问题、根因分析、修复措施和验证结果。目标是把一次性错误沉淀为系统改进，让下一批次更顺畅。

## 历史问题库

### 2026-07-01 发布复盘

#### 问题 A：回答被折叠（AI检测）
- **现象**: 回答发布后短时间内被系统折叠，提示疑似 AI 生成内容。
- **根因**:
  - 回答字数过长（800-1500字）
  - 回答配有 AI 生成图片
  - 结尾整段 WADESK 宣传 + 链接
  - 段落过于工整，缺少口语化和不确定性表达
- **修复措施**:
  - `content_config.json`：回答字数限制改为 **300-800字**
  - 回答不配图片，纯文字
  - WADESK 植入可选，仅在自然衔接时轻提一句，禁止放链接
  - 新增反 AI 检测规则：段落长短不齐、口语化表达、不确定表达、具体案例
- **状态**: 🟢 已修复
- **验证**: 待 2026-07-02 发布后检查是否仍被折叠

#### 问题 B：回答与问题货不对板（最严重）
- **现象**: 回答被发布到与外贸无关的问题下，如硬件测试工程师、社会现象问题。
- **根因**:
  - `question_bank.json` 中多个问题标题与 URL 不匹配
  - 635678901、665678901 被记录为外贸问题，实际是其他话题
  - 脚本使用了 `verified=false` 的问题
- **修复措施**:
  - 删除 635678901 和 665678901
  - 修正 555079740 的标题
  - `pick_questions.py` 强制只使用 `verified=true` 的问题
  - 新增 `invited_questions.json`，优先使用知乎邀请回答的真实问题
- **状态**: 🟢 已修复
- **待办**: 手动删除已发布到错误问题的回答（665678901、635678901）

#### 问题 C：内容重复
- **现象**: 3 条回答都围绕"客户管理/防飞单/客户不跟丢"同一主题，缺乏多样性。
- **根因**: 选题时未强制要求文章与回答的 pillar 错开。
- **修复措施**:
  - 自动化 prompt 要求：文章和回答的 pillar 必须错开
  - `content_config.json` 增加 pillar 覆盖检查提示
- **状态**: 🟢 已修复
- **验证**: 待 2026-07-02 检查选题分布

#### 问题 D：脚本 bug - `best_h` 未初始化
- **现象**: `publish_answer_v11.py` 在编辑器未加载时崩溃：`UnboundLocalError: best_h`
- **根因**: 循环内只在 `count > 0` 时赋值 `best_h`，未处理 0 个编辑器的情况。
- **修复措施**: 在循环前添加 `best_h = 0` 初始化。
- **状态**: 🟢 已修复

#### 问题 E：URL 检测假阴性
- **现象**: 回答实际发布成功，但 `publish_log.json` 记录的 URL 不正确或包含 `/write`。
- **根因**: 知乎回答提交后页面可能未立即跳转到 `/answer/xxx`，脚本只检测了 `/answer/` 路径。
- **修复措施**:
  - `daily_publish_v2.py` 的 `[LIKELY OK]` 逻辑提取表单消失后的 URL
  - 增加对 `/question/{qid}/write` 和 `/question/{qid}` 的兼容
- **状态**: ⚪ 已观察
- **待办**: 持续观察 2026-07-02 的 URL 记录是否准确

## 运行复盘脚本

```bash
cd zhihu_auto
python post_publish_review.py
```

脚本会：
1. 读取 `publish_log.json` 检查发布成功/失败
2. 检查内容文件是否被正确发布
3. 生成本次复盘草稿
4. 追加到 `POSTMORTEM.md`
